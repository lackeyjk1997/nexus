## Universal gaps (do NOT repeat per route)

Every route below shares these structural gaps (per DECISIONS.md 2.1):
- **No authentication.** No route checks a session, token, or user identity. `teamMembers.id` / `observerId` / `memberId` / `aeId` are passed in the request body or query string by the client, and the route trusts them. There is no RLS backing this up either (see 02-SCHEMA.md).
- **No rate limiting.** Any caller can hit any AI endpoint at any rate.
- **No CORS policy.** Only `/api/mcp` sets CORS headers. Everything else accepts cross-origin based on Next.js defaults.
- **No structured request validation.** Most routes do ad-hoc `if (!foo) 400` checks; none use Zod or a schema library (one exception: `/api/mcp` uses Zod for tool inputs).
- **Errors are caught and downgraded.** Many routes wrap work in `try/catch` and return 500 with a generic message, burying the cause in `console.error`. Some deliberately return 200 with empty data on failure (`/api/deals/close-analysis`, `/api/field-queries`).

Per-route "Known issues" sections below flag issues _beyond_ these universal gaps.

## Universal conventions

- `export const dynamic = "force-dynamic"` appears on all but one route (the one exception: `/api/playbook/ideas/[id]` exports it but at the bottom; still applied).
- Data layer: `drizzle-orm` queries via `db` from `@/lib/db`. Every write is a direct SQL write — no domain services, no repository layer.
- AI client: `new Anthropic()` or `new Anthropic({ apiKey })` constructed inline per request. No shared client. Model everywhere: `claude-sonnet-4-20250514`.
- Many routes trigger internal work by `fetch(appUrl + "/api/...")` instead of invoking shared functions. This causes a network hop within a Vercel serverless invocation — and bumps into function timeouts when the callee is itself slow.

- **Rivet:** none directly; indirectly via internal fetch to `/api/deal-agent-state` for agent memory.

  - Internal fetch to `/api/deal-agent-state?dealId=...` (line 485) inside a serverless function is a same-process network hop that costs latency.

## `POST /api/deal-fitness/analyze`
- **File:** [deal-fitness/analyze/route.ts](../../apps/web/src/app/api/deal-fitness/analyze/route.ts) — 989 LOC
- **Methods:** POST
- **Purpose:** The oDeal fitness analyzer. Gathers every transcript + every email activity for a deal, builds a chronological timeline, and sends it to Claude with a 250-line system prompt defining 25 inspectable buyer behaviors across 4 fit categories (business / emotional / technical / readiness). - **Tables written:** `dealFitnessEvents` (delete-then-insert all 25), `dealFitnessScores` (upsert)
- **Claude API calls:** 1 (call site #15). `max_tokens: 16000`.
- **Rivet:** none
- **maxDuration:** 300
- **Known issues:**
  - `readnessFitDetected` typo carried through here (see 02-SCHEMA.md §4.2).
  - JSON parse has 3 fallback strategies — which is robust but tells you how unpredictable the model output is at this size.
  - Uses Claude's `buyingCommitteeExpansion.contacts[].role` straight through without mapping; `stakeholderEngagement.departmentsEngaged` is count of distinct `role` values, which conflates roles and departments.
  - `buyerMomentum.commitmentFollowThrough.commitments[].week = 0` (line 858) — week tracking isn't actually computed, it's stubbed.

- **Request shape:** `{ dealId: string, transcriptText: string, transcriptId?: string }`
- **Response shape:** `{ status: "processing", dealId }`
- **Tables read:** `deals`, `companies`, `meddpiccFields`, `contacts`, `agentConfigs`, `playbookIdeas`, `teamMembers`
- **Tables written:** `callTranscripts` (sets pipelineProcessed=true)
- **Claude API calls:** none directly (pipeline actor makes 6+ Claude calls once it runs).
- **Rivet:** `transcriptPipeline.getOrCreate([dealId]).send("process", {...})`
- **maxDuration:** 300
- **Known issues:** The `maxDuration = 300` is just for the enqueue call which should return in < 1s — likely a copy-paste from the Rivet route's config. The pipeline runs async in Rivet, not in this function.

### Notable cross-route risks
- **Multiple stage-change paths:** `/api/deals/stage` always writes `changedBy="human"` + stage history + activity + observations. `/api/deals/[id]/update` writes the `stage` column directly with NO history, NO activity, NO observations. A stage transition via `/update` silently bypasses the audit.
- **Uncoordinated `getEffectiveType()` writes:** `/api/agent/save-to-deal`, `/api/deals/[id]/meddpicc-update`, `/api/field-queries/respond` all write `type: "note_added"` with `metadata.source` — perpetuating the schema debt from 02-SCHEMA.md §4.1 every time they run.
- **CLAUDE.md claim about landing-page reset is stale.** The landing page at `apps/web/src/app/page.tsx` has no reference to `/api/demo/reset`. Reset is only callable from the `AdminReset` component triggered via a `window.dispatchEvent("admin-reset-trigger")` from the sidebar (line 119 of `sidebar.tsx`).
- **Internal same-process fetches:** `/api/agent/call-prep` fetches `/api/deal-agent-state`; the coordinator actor fetches `/api/intelligence/persist-pattern`; `/api/analyze/link` and `/api/demo/prep-deal` both fetch `/api/transcript-pipeline`; the pipeline actor fetches `/api/deals/[id]/meddpicc-update` + `/api/observations` + `/api/playbook/ideas/[id]` + `/api/deal-agent-state` + `/api/agent/call-prep`; `/api/mcp` fetches `/api/agent/call-prep` + `/api/observations`. Every hop is a full HTTPS roundtrip (even in dev on localhost) and a fresh serverless invocation on Vercel, multiplying latency and cold-start exposure.
- **Sarah Chen as the implicit current user:** `/api/mcp` hardcodes her for both call-prep generation and observation logging. `/api/book` requires an `aeId`. `/api/deals/close-analysis` uses "first team member" as a last-resort fallback.
